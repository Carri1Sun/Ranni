# 数据报告 / 洞察页参考资料

## 来源

- Nielsen Norman Group How People Read Online: https://www.nngroup.com/articles/how-people-read-online/
- web.dev Web Vitals: https://web.dev/articles/vitals

这些 URL 仅用于人工溯源。以下内容已经把来源思路本地化，agent 运行时不要访问外部 URL。

## 本地化来源笔记

### NN/g How Users Read on the Web

- 数据报告读者通常先扫描结论、数字、图表标题和图注，再决定是否深入阅读方法和章节。
- 页面应先给核心结论和关键数字，再给图表证据、分析和方法说明。
- 小标题要写出发现，例如“移动端转化差距扩大”，避免只写“图表一”或“关键数据”。
- 每个图表需要一句结论、口径、时间范围和必要图例。图形不能替代解释。

### web.dev Web Vitals

- 数据页常有大图表、图片和交互组件，需要关注加载、交互和视觉稳定性。
- LCP 要求首屏主要内容尽快出现；核心结论和关键数字应优先渲染，不被大图表阻塞。
- CLS 要求布局稳定；图表和图片必须预留尺寸，避免加载后推动正文和 CTA。
- INP 要求交互反馈及时；筛选、切换、展开和下载入口不要被复杂图表拖慢。

## 可提炼设计原则

- 数据洞察页应先给结论，再展开证据。读者需要先知道发生了什么、为什么重要、影响谁。
- 关键数字需要上下文：单位、时间范围、样本、对比基准、变化方向和含义应靠近数字出现。
- 每个图表都需要一句结论和数据口径。图形本身不能替代解释。
- 方法说明是信任来源。样本、来源、采集时间、限制和定义需要透明。
- 数据密集页面也要关注性能和稳定性。首屏不要被大图表阻塞，图表容器要有稳定尺寸。

## 结构建议

- 报告标题、时间范围和核心结论。
- 2 到 4 个关键数字，配简短解释。
- 图表模块：结论、图表、口径、图例。
- 分章节分析：按问题组织，而不是按数据表顺序。
- 方法说明：样本、来源、时间、限制。
- 下载、分享或订阅 CTA。

## 常见失误

- 一开始展示大量图表，读者看不到结论。
- 数字缺少单位和时间范围。
- 图表没有图例、口径或解释。
- 移动端复杂图表难读，没有摘要替代。
